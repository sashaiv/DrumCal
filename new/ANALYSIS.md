## 2. Что хорошо в каждом треке

### Трек A — сильные стороны

- **Полный конвейер** — от заказа до выходных барабанов (5 шагов)
- **Excel I/O** — читает из `input.xlsx`, пишет профессиональный отчёт
- **Склад готового кабеля** — покрывает отрезки журнала из остатков
- **Склад изолированных жил** — multi-spool поддержка (смена катушки между сегментами)
- **Детальные ошибки/предупреждения** — П1–П9 параметры
- **Выходные барабаны** — greedy bin-packing для распределения отрезков
- **Чистая архитектура** — модули изолированы, легко тестировать

### Трек B — сильные стороны

- **OR-Tools CP-SAT** — математически оптимальное решение (не жадный алгоритм)
- **Гибкие кабели** (`flexible=True`) — поглощение остатков барабана ТПЖ
- **Группировка по составу жил** — отдельный FFD для каждой группы (5ж, 4ж, 3ж)
- **Оптимальный порядок цветов** — перебор перестановок (≤8 цветов) или NN-жадный
- **Обратный порядок для скрутки** — изолировщик мотает снизу вверх, скрутчик разматывает сверху вниз
- **Авто-палитра** — неизвестные цвета получают фоновый цвет автоматически

## 3. Проблемы и пробелы

### Проблема 1 — Жадный алгоритм распределения ТПЖ (критично)

В `planner.py:_allocate_cores_for_batch` выбирается наибольший доступный барабан:
```python
best_rw = max(rw_candidates, key=lambda r: r.available)
```

Это **может дать плохой результат**. Пример:
- Барабан1 = 3000 м, Барабан2 = 2500 м
- Партия1/Красный: нужно 1500 м; Партия1/Синий: нужно 1500 м
- Партия2/Зелёный: нужно 2400 м
- Итого нужно: 5400 м < 5500 м доступных → задача решаема!

**Жадный**: Красный берёт Барабан1 (3000→1500), Синий берёт Барабан2 (2500→1000),
Зелёный нужно 2400 м → Барабан1=1500<2400, Барабан2=1000<2400. **Ошибка!**

**Оптимальный** (CP-SAT): Красный + Синий → оба с Барабана1 (3000→0),
Зелёный → Барабан2 (2500→100 м остаток). ✓

### Проблема 2 — Трек B не имеет полного конвейера

`claude_insulation_v4.ipynb` решает только задачу распределения ТПЖ по кабелям.
Нет: склада кабеля, склада изолированных жил, выходных барабанов, Excel-ввода v5.

### Проблема 3 — Трек A не поддерживает `flexible` кабели

Нет логики "поглощения остатков" — когда барабан почти пустой и есть гибкий кабель,
который может взять лишние метры.

### Проблема 4 — FFD не учитывает группы по составу жил

В Треке A FFD пакует все отрезки вместе. Проблема: если заказ содержит кабели 5ж и 4ж,
то наборы для скрутки будут смешаны → нельзя скрутить (разный состав цветов).
Трек B решает это правильно: отдельный FFD по группам.

### Проблема 5 — Входной формат v5 не поддерживается в Треке A

`parser.py` читает старый формат. `input_v5.xlsx` добавляет:
- Колонку `Гибкий (ДА/НЕТ)` в заказах
- Поле `Кабельный журнал` (строительные длины через запятую)
- Лист `3. Барабаны ТПЖ` вместо раздела в П-Ф

### Проблема 6 — Нет инструкции с обратным порядком для скрутчика

Трек A генерирует инструкцию в порядке намотки. Но скрутчик разматывает в обратном порядке
(снизу вверх). Трек B корректно инвертирует порядок.

## 5. Детали реализации — CP-SAT в Шаге 4

### Постановка задачи

После шага 3 у нас есть список **задач**: `(batch_id, color, raw_needed)` —
каждой задаче нужно назначить ровно один барабан ТПЖ.

Ограничение `MAX_SPLITS = 1` (запрет стыков жилы) делает это задачей **назначения**:
каждая задача → ровно один барабан.

### CP-SAT модель

```
Переменная:  assign[task, drum] ∈ {0, 1}
             assign[task, drum] = 1 → задача берёт raw_needed метров с drum

Ограничения:
  1. Для каждой задачи: Σ assign[t, d] == 1            (ровно один барабан)
  2. Для каждого барабана: Σ assign[t, d] * raw_needed[t] <= drum.available
  3. (Если MAX_SPLITS > 1): разбить задачу на несколько барабанов

Цель: минимизировать отходы (максимизировать использование)
```

### Почему CP-SAT лучше жадного

| Критерий | Жадный | CP-SAT |
|----------|--------|--------|
| Оптимальность | Не гарантирует | Математически доказана |
| Обнаружение неразрешимости | Может ошибиться | Точно (INFEASIBLE статус) |
| Гибкие кабели (поглощение остатков) | Нет | Встроено в модель |
| Скорость | O(n²) мгновенно | 1–30 сек для сотен задач |

## 6. Детали — FFD с группировкой (Шаг 3)

Проблема: кабели с разным составом жил нельзя скручивать вместе.

Решение (из Трека B): **группировать по frozenset(colors)**, затем FFD внутри каждой группы.

```python
groups = {}
for batch in batches:
    key = frozenset(batch.colors)
    groups.setdefault(key, []).append(batch)
# FFD per group → отдельные наборы Н1, Н2, …
```

Метки групп: `5ж`, `4ж`, `3ж`, `2ж`. При коллизиях: `4ж-А`, `4ж-Б`.

## 7. Детали — гибкие кабели (`flexible=True`)

Гибкий кабель = позиция заказа, длина которой **может быть увеличена** для поглощения
остатков барабана. Физически: сдача «максимально возможной длиной».

В CP-SAT модели:
- Негибкий: `Σ x[batch, d] == min_length` (равенство)
- Гибкий:   `Σ x[batch, d] >= min_length` (неравенство)

Flexible-кабели получают дополнительный материал в конце каждого барабана.

## 10. Глоссарий (для справки)

| Термин | Значение |
|--------|---------|
| ТПЖ | Токопроводящая жила — неизолированный провод на входном барабане |
| Партия (ПС) | Партия скрутки — один прогон скрутки, несколько строительных длин |
| Набор (Нi) | Комплект мотков по всем цветам для одного прохода скрутки |
| П/Ф | Полуфабрикат — изолированная жила (из изолировки или со склада) |
| ИЗ | Прогон изолирования — один цвет, один барабан ТПЖ → один моток |
| FFD | First-Fit Decreasing — алгоритм упаковки отрезков в партии |
| CP-SAT | Constraint Programming - SAT solver (OR-Tools) |
| Строительная длина | Длина одного отрезка кабеля из кабельного журнала |
| Гибкий кабель | Позиция заказа, длина которой может быть увеличена |
| Startup loss (П1) | Потери ТПЖ на заправку изолировочной линии (~5 м/прогон) |
| Tolerance (П4) | Запас на обрезку торцов (~3 м/отрезок) |
