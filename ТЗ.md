# Техническое задание: Length Helper — планировщик производства кабеля

## 1. Назначение системы

Программа автоматически составляет **производственный план** для выпуска многожильного скрученного кабеля.
На входе — кабельный журнал (список нужных отрезков), остатки склада, параметры оборудования.
На выходе — пошаговые инструкции для цеха: что изолировать, что скручивать, на какой барабан намотать.

---

## 2. Входные данные

### 2.1 Склад изолированных жил
Каждая позиция:
- Идентификатор, название
- Цвет (Синяя, Натуральная, Чёрная, …)
- Сечение + тип (2,5ок, 25мк, …)
- Материал изоляции (LS, HF, …) и признак огнестойкости (FR)
- **Список длин** — несколько физических катушек под одним цветом/типом

### 2.2 Склад ТПЖ (неизолированных жил)
- Идентификатор, название
- Сечение + тип
- Длина барабана

### 2.3 Склад готового кабеля
- Марка кабеля
- Длина остатка

### 2.4 Заказ
- Марка кабеля (ВВГ 4×25мк(N) и т.п.)
- Общая длина или кабельный журнал (список отрезков)
- Состав жил: цвета, сечение, тип, материал изоляции, огнестойкость

### 2.5 Параметры технологического процесса
| Параметр | Значение по умолчанию | Пояснение |
|---|---|---|
| max_insulation_run | 4500 м | Макс. прогон изолировочной линии |
| max_twisting_run | 2100 м | Макс. партия скруточного станка |
| min_construction_length | 100–450 м | Мин. строительная длина кабеля |
| allow_splicing | False | Разрешить/запретить спайки |

---

## 3. Производственные сценарии

### Сценарий A. Скрутить по заказу — всё из склада жил

**Условие:** изолированные жилы всех цветов нужного типа есть на складе в нужном количестве.
**Ограничений по изолированию нет** — ТПЖ не используется.

Алгоритм:
1. Построить журнал отрезков.
2. Покрыть что можно из склада готового кабеля.
3. Упаковать оставшиеся отрезки в партии скрутки (FFD bin-packing, ≤ max_twisting_run).
4. Для каждой партии и каждого цвета — выделить катушку(и) со склада жил.
   - Одна катушка покрывает партию целиком → 1 запись.
   - Нет одной катушки достаточной длины, но суммарно хватает → **смена катушки между отрезками**, несколько записей (П/Ф №1, П/Ф №2 …).
5. Распределить все отрезки журнала по выходным барабанам.

**Выходные документы:** таблица заправок (ТАБЛИЦА ЗАПРАВОК), реестр П/Ф, план намотки на барабаны.

---

### Сценарий B. Скрутить максимум из имеющегося

**Условие:** заказ задан с журналом, но склад жил **не покрывает весь журнал**.
Цель: произвести **максимально возможную длину**, не используя спайки.

Ключевые правила:
- Плановая длина отрезка уменьшается до минимума из доступных катушек по всем цветам.
- Если урезанная длина ≥ min_construction_length → принять на отдельный барабан с пометкой.
- Если урезанная длина < min_construction_length → отрезок **пропускается** (предупреждение).
- **Спайка запрещена** — жила должна идти непрерывно от одной катушки.
- Остатки катушек, слишком короткие для любого отрезка журнала, **уходят в отход** автоматически.

> **Пример:** журнал [670, 711, 1087, 1328], склад Синей = [2839, 829, 439].
> После ПС-001 (2039м) на Синей остаётся катушка 829м.
> Сегмент 1087м → урезается до 829м. Принимается как отдельный отрезок.
> Катушка 439м — в отход (< min_construction_length для любого оставшегося сегмента).

---

### Сценарий C. Скрутить по заказу — с изолированием

**Условие:** на складе жил нужного цвета/типа **нет или не хватает**;
есть барабаны ТПЖ (неизолированная жила).

Алгоритм (шаг 4 планировщика):
1. Проверить склад изолированных жил (приоритет 1).
2. Если нехватка по одному или нескольким цветам → запланировать **прогон изолирования** с ТПЖ.
   - Один прогон = одна партия скрутки × один цвет.
   - Длина прогона ≤ max_insulation_run.
   - Для прогона выбирается наименьший барабан/катушка из таблицы ёмкостей.
3. Если ни ТПЖ, ни склада жил нет → ошибка.

**Дополнительные ограничения на изолирование:**
- Один прогон изолирования = один непрерывный кусок без спайки ТПЖ.
- Если нужная длина > max_insulation_run → ошибка (нужно делить на партии скрутки меньшего размера).

---

### Сценарий D. Скрутить максимум имеющегося — с изолированием

**Условие:** часть цветов из склада жил, часть — из ТПЖ с изолированием;
склад жил не покрывает весь журнал.

Алгоритм — комбинация B и C:
1. Для каждого отрезка партии: определяем доступный максимум по складским цветам.
2. Фактическая длина = **min(склад_max по всем складским цветам)**.
3. Если фактическая длина < плановой → урезаем весь отрезок.
4. Изолировочные прогоны рассчитываются уже по фактической длине.

---

## 4. Обрезки и отходы

### 4.1 Откуда берутся отходы

| Ситуация | Как возникает | Что делать |
|---|---|---|
| Остаток катушки мал | После выделения жилы под партию остаток < min_construction_length | Списать в отход; в реестре остатков показать с пометкой |
| Обрыв/повреждение при изолировании | Начали прогон, порвалась жила, получилось меньше запланированного | Длина прогона корректируется вручную; переплановать партию скрутки с фактической длиной |
| Потери в цехе (брак, намотка) | Кабель принят короче из-за обрезки дефекта | Фактическая длина барабана ≠ плановой; отражается в отчёте об остатках |
| Неперекрывающийся остаток ТПЖ | После изолирования остаток ТПЖ слишком короткий для нового прогона | Остаток фиксируется на складе; при следующем расчёте будет предложен если хватает |

### 4.2 Принцип «выкрутить максимум»

Если катушка не покрывает плановый отрезок целиком:
- Урезаем **только данный отрезок** — соседние не трогаем.
- Урезанный отрезок **всегда идёт на отдельный барабан** (смешивание не допускается).
- В ТАБЛИЦЕ ЗАПРАВОК для урезанного отрезка указывается его фактическая длина.
- Остаток катушки после урезанного отрезка используется для следующего сегмента (если хватает).

### 4.3 Что никогда не допускается
- **Спайка жил** внутри отрезка — не допускается (allow_splicing = False по умолчанию).
- **Смешивание отрезков с разных партий** на одном барабане.
- **Смешивание складского и производственного кабеля** на одном барабане.

## 5. Алгоритм планирования (пошагово)

```
Для каждого заказа:

  Шаг 1. Журнал
    • Если журнал задан — использовать.
    • Если нет — разбить total_length на равные части ≤ max_twisting_run.

  Шаг 2. Покрытие из склада готового кабеля
    • Для каждого отрезка журнала — найти остаток на складе той же марки.
    • Выбрать наименьший подходящий остаток (минимизация потерь).
    • Непокрытые отрезки → в производство.

  Шаг 3. Упаковка в партии скрутки (FFD)
    • Отсортировать производственные отрезки по убыванию.
    • Упаковать в партии с суммой ≤ max_batch (min из max_twisting_run и макс. ёмкости барабана жилы).
    • Отрезок > max_batch → ошибка (требует спайки).

  Шаг 4. Выделение жил (всё цвета одновременно, посегментно)
    Для каждого отрезка партии:
      а) Для каждого цвета — определить максимально возможную длину:
         - Есть открытая катушка с достаточным остатком → max = planned_seg.
         - Есть другая катушка ≥ planned_seg → max = planned_seg (новая катушка).
         - Нет катушки ≥ planned_seg, но есть ≥ min_len → max = best_available.
         - Вообще нет ничего ≥ min_len → max = 0.
      б) Фактическая длина = min(max по всем складским цветам).
      в) Если фактическая < min_construction_length → пропустить отрезок (предупреждение).
      г) Если фактическая < плановой → урезать (предупреждение).
      д) Выделить фактическую длину из каждой катушки.
         Смена катушки → зафиксировать предыдущую, начать новую.
      е) Для цветов без склада → запланировать прогон изолирования.

  Шаг 5. Выходные барабаны
    • Складские отрезки → каждый на отдельный барабан.
    • Производственные отрезки одной партии → упаковать вместе (greedy bin-packing).
    • Отрезки из разных партий → разные барабаны.
```

## 7. Граничные случаи и поведение системы

| Ситуация | Поведение |
|---|---|
| Журнал не задан | Автоматически делим на отрезки ≤ max_twisting_run |
| Сумма журнала ≠ total_length | Предупреждение, работаем дальше |
| Отрезок журнала > max_twisting_run | Ошибка (спайка запрещена) |
| Нет ни жил, ни ТПЖ для цвета | Ошибка |
| Одна катушка на несколько отрезков | Смена катушки между отрезками; → П/Ф №1, П/Ф №2 в таблице заправок |
| Несколько катушек одного цвета | Отдаём наименьшую подходящую (экономия) |
| Остаток катушки < min_construction_length | В отход; в план не включается |
| Урезанный отрезок < min_construction_length | Пропускается; предупреждение |
| Нет таблицы ёмкости барабанов для марки | Предупреждение; барабаны не распределяются |
| Складской отрезок кабеля > макс. барабана | Предупреждение; намотка на несколько барабанов |

## 8. Что **не входит** в текущую область

- Оперативный учёт (ввод фактических длин по завершении работы).
- Корректировка плана при отклонениях в ходе производства.
- Автоматический выбор марки кабеля (принимается от оператора).
- Расчёт времени и загрузки оборудования.
- Учёт потерь на зачалку/отрезание концов (учитываются внешне через коэффициент).

## 9. Что **не входит** в текущую область (без изменений)

- Оперативный учёт (ввод фактических длин по завершении работы) — **кроме п.10.3**.
- Расчёт времени и загрузки оборудования.
- Автоматический выбор марки кабеля (принимается от оператора).

## 10. Выявленные проблемы и план доработок

> Анализ пути «начальник производства: от получения заказа до сдачи».

### Приоритет: ВЫСОКИЙ

#### П1. Потери на заправку изолировочной линии
**Проблема:** Каждый прогон изолирования начинается с «заправки» — первые 3–10 м жилы
прогоняются вхолостую до стабилизации натяжения и режима экструдера. Это **системная потеря
на каждый прогон**. Алгоритм планирует ровно нужную длину, не учитывая эти потери.

**Следствие:** Если барабан ТПЖ содержит ровно нужную длину — реальный прогон выйдет короче
и партия скрутки окажется невыполнимой. Это не ошибка алгоритма, а ошибка реального производства.

**Решение:** Параметр `insulation_startup_loss_m` (по умолчанию 5 м). Расход ТПЖ
= длина прогона + потери на заправку. В выходных документах показывать оба числа:
«Длина прогона: 1500 м / Снять с барабана: 1505 м».

**Статус:** ✅ Реализовано.

---

#### П2. Объединение прогонов изолирования между заказами
**Проблема:** Если два заказа одновременно требуют, например, Синюю LS 2,5ок — алгоритм
планирует два отдельных прогона (по 500 м и по 800 м). Реальный начальник производства
сделает один прогон 1300 м, сэкономив одну наладку машины.

**Следствие:** Лишние переналадки оборудования, лишние потери на заправку, менее
эффективная загрузка изолировочной линии.

**Решение:** Двухфазный расчёт: сначала планируем партии скрутки для всех заказов,
затем агрегируем потребности в изолировании одного цвета/типа/материала, объединяем
в один прогон если суммарно ≤ max_insulation_run. Требует рефакторинга `plan()`.

**Статус:** ⬜ Не реализовано (архитектурно сложно, низкий риск ошибки при планировании
одного заказа за раз).

---

#### П3. Обратная связь план → факт
**Проблема:** Оператор изолировочной линии может получить меньше, чем планировалось
(обрыв жилы, дефект барабана). Сейчас нет механизма ввода фактической длины и
автоматического пересчёта партии скрутки.

**Следствие:** Мастер вынужден вручную пересчитывать, что скрутить из фактически
полученной жилы.

**Решение:** Режим «корректировка» — оператор вводит фактическую длину прогона,
система пересчитывает, какие отрезки партии скрутки выполнимы (применяя логику
«выкрутить максимум»).

**Статус:** ⬜ Не реализовано (требует интерактивного интерфейса).

---

### Приоритет: СРЕДНИЙ

#### П4. Запас на строительную длину (обрезка торцов)
**Проблема:** В кабельном журнале отрезок «670 м» — это минимальная длина для заказчика.
При намотке и обрезке торцов реально расходуется чуть больше (2–5 м на отрезок).
Алгоритм планирует ровно 670 м потребления жилы, из-за чего по факту кабель может
оказаться 668–669 м, что не пройдёт ОТК.

**Решение:** Параметр `length_tolerance_m` (по умолчанию 3 м). Для каждого сегмента
партии скрутки прибавляется к длине прогона изолирования (N отрезков × tolerance).
В выходных документах — пояснение, что длины плановые (минимум), с учётом запаса.

**Статус:** ✅ Реализовано (для прогонов изолирования).

---

#### П5. Сохранение порядка журнала в партиях скрутки
**Проблема:** FFD-упаковка сортирует отрезки по убыванию длины. В результате физический
порядок отрезков на барабане не совпадает с нумерацией кабельного журнала.
Для заказчиков, принимающих барабаны по порядку (объект А, объект Б), это важно.

**Решение:** Параметр `keep_journal_order: bool` (по умолчанию False).
Если True — отрезки упаковываются в партии в порядке журнала без сортировки FFD.

**Статус:** ✅ Реализовано.

---

#### П6. Предупреждение «остаток катушки → отход»
**Проблема:** После использования катушки остаток может составить, например, 12 м —
меньше минимальной строительной длины любого будущего заказа. Де-факто это отход.
Сейчас алгоритм фиксирует этот остаток как «доступный» в списке остатков.

**Решение:** Параметр `waste_warning_threshold_m` (по умолчанию 50 м). Если остаток
после использования катушки < порога — добавляется предупреждение «катушка X: остаток
Yм — вероятно в отход». В реестре остатков такие позиции помечаются особо.

**Статус:** ✅ Реализовано.

---

#### П7. Межзаказная конкуренция за ресурс
**Проблема:** Два заказа претендуют на одну катушку жилы. Алгоритм обрабатывает заказы
последовательно и просто «первому — всё», второй получает ошибку.

**Следствие:** Нет явного контроля приоритетности заказов. Оператор не видит, что
второй заказ пострадал из-за первого.

**Решение:** Добавить поле `priority: int` в `CableOrder`. При нехватке ресурса —
явное предупреждение «ресурс X использован заказом Y (приоритет Z), заказу W не хватило».

**Статус:** ⬜ Не реализовано.

---

### Приоритет: НИЗКИЙ

#### П8. Стратегия «пристроить хвосты»
**Проблема:** Есть несколько коротких катушек (371 м, 439 м), которые не подходят ни
под один отрезок журнала текущего заказа, но могли бы подойти для другого, более
короткого заказа. Алгоритм не ищет «куда пристроить» остатки.

**Решение:** После планирования всех заказов — отдельный отчёт «Непристроенные остатки»
с указанием максимальной длины применения. Это позволит плановику вручную принять решение.

**Статус:** ⬜ Не реализовано.

---

#### П9. Остаток ТПЖ после прогона
**Проблема:** После изолирования на барабане ТПЖ остаётся некоторый остаток.
Алгоритм его считает и показывает в «Остатках». Но не отображает явно:
«С барабана снято X м, ушло в прогон Y м, потери на заправку Z м, остаток W м».

**Решение:** Детализация в листе «Прогоны изолирования» — отдельная колонка
«Расход ТПЖ» = длина прогона + потери на заправку.

**Статус:** ✅ Реализовано (через поле `raw_wire_consumed` в `InsulationRun`).

---

сначала (изолирование), что потом (скрутка), в каком порядке партий.

**Решение:** Добавить первый лист «ПОСЛЕДОВАТЕЛЬНОСТЬ РАБОТ» с пошаговым списком:
1. Прогон изолирования ИЗ-001 (до этого шага ПС-001 невозможна)
2. Партия скрутки ПС-001
3. Прогон изолирования ИЗ-002
4. Партия скрутки ПС-002
…

**Статус:** ⬜ Не реализовано.
